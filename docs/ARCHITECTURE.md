## 1. What the system is

An internal Applicant Tracking System (ATS) for a Philippine staffing agency. Candidates move through a linear pipeline; each HR department logs in to its own role and sees only the applicants currently at its stage. Once an applicant is deployed, they graduate into an **Employee Masterlist** managed by Employee Relations.

```
Applicant ─► Screening ─► Assessment ─► Selection ─► Medical/Requirements ─► Deployment ─► Employee Relations (Masterlist)
```

The whole workflow is driven by a single `status` string on one wide `recruitment_applicants` row. Each module "owns" a set of statuses and filters the shared applicant list down to those.

## 3. Runtime topology

```
Browser (Vite dev :5173)
   │  fetch('/api/...')  ← relative paths only
   ▼
Vite dev-server proxy  (vite.config.ts)
   │  /api → http://localhost:5001
   ▼
Express API (:5001)  ── mysql2 pool ──►  MySQL  company_ftl
```

- The frontend **only ever calls relative `/api/...` URLs**. There is no `VITE_API_BASE_URL` in the fetch path; everything relies on the Vite proxy in dev. In production the API must be reverse-proxied under the same origin at `/api`.
- **Port 5001, not 5000.** macOS Control Center / AirPlay Receiver occupies 5000, so the backend runs on 5001 (`server/.env`) and the Vite proxy points there. See [DEVELOPMENT.md](DEVELOPMENT.md).

## 4. Backend architecture

Layered Express app: **routes → controllers → models → MySQL pool**.

```
server/
├── index.js                 # app wiring, CORS, route mounting, listen(5001)
├── config/
│   ├── index.js             # env → { PORT, DB_* }
│   └── roles.js             # ROLES enum (0..5)
├── middleware/
│   ├── auth.js              # authRequired, requireRoles, JWT_SECRET  ⚠ defined but UNUSED
│   └── logger.js            # request logger
├── routes/                  # thin: map HTTP verb → controller fn
│   ├── auth.js              # /api/auth   (login, register)
│   ├── applicants.js        # /api/applicants  (+ *-history endpoints)
│   ├── principals.js        # /api/principals
│   ├── clients.js           # /api/clients   ⚠ legacy alias → principals
│   └── masterlist.js        # /api/masterlist
├── controllers/             # request parsing, field mapping, business logic
│   ├── applicantsController.js
│   ├── principalsController.js
│   ├── clientsController.js  # re-exports principalsController (compat shim)
│   └── masterlistController.js
└── models/                  # SQL + schema self-provisioning
    ├── applicant.js         # recruitment_applicants + 4 history tables
    ├── principal.js         # principals
    ├── applicantPrincipal.js# applicant_principals (junction)
    ├── applicantClient.js   # re-exports applicantPrincipal (compat shim)
    ├── client.js            # pool only (compat shim)
    ├── masterlist.js        # masterlist_employees (+ self-migration & sync)
    └── user.js              # users
```

### Key backend behaviors
- **Self-provisioning schema.** Models call `ensureTables()` / `ensureUserTable()` / `ensureTable()` at the top of nearly every query — they run `CREATE TABLE IF NOT EXISTS` (and, for masterlist, `ALTER TABLE` migrations) on each call. The DB bootstraps itself; there is no migration runner. See [IMPROVEMENTS.md](IMPROVEMENTS.md) (hot-path cost, schema drift).
- **`masterlist.js` is self-migrating.** `migrateTable()`, `columnExists()`, `backfillPrincipalFromApplicants()` evolve the table in place; `syncDeployedApplicants()` (`POST /api/masterlist/sync`) pulls deployed applicants into the masterlist; FLI numbers are generated by `getNextFliNumber()` / `formatFliNumber()`.
- **Upsert semantics.** `upsertRecruitmentApplicant()` updates by `applicant_no` with `col = IFNULL(?, col)`, so empty/undefined values **preserve** the existing DB value (you can set but not clear a field through this path).
- **No auth enforcement.** Every `/api/applicants`, `/api/principals`, `/api/masterlist` route is currently public — the JWT middleware exists but is not applied. See [IMPROVEMENTS.md](IMPROVEMENTS.md) §Security.

## 5. Frontend architecture

```
src/
├── main.tsx / App.tsx        # entry + top-level "router" (state switch, no react-router)
├── Global/
│   ├── Sidebar.tsx           # nav; filters items by role
│   ├── ApplicantSidebar.tsx  # 950+ line applicant detail/edit drawer (shared across modules)
│   └── NavigationContext.tsx # activeSection / currentApplicantNo context
├── api/                      # TypeScript interfaces + a few fetch helpers
│   ├── applicant.ts, applicantPersonal.ts, applicantDocuments.ts
│   ├── assessmentStatus.ts   # ApplicationStatus union + status sets
│   ├── principal.ts, client.ts (legacy), masterlist.ts
├── components/               # LoadingScreen, Filters/
└── pages/
    ├── dashboard, screening, assessment, selection,
    │   engagement, employee_relations, recruitment-database (thin page shells)
    ├── auth/ (Login, Register)
    └── components/<Module>/
        ├── hooks/    useXxxApplicants.ts   # data fetch + state + mutations
        ├── utils/    xxxUtils.ts           # status sets, row mappers, formatters
        └── components/ tables, toolbars, history tables, modals
```

### Navigation
- **No React Router.** `App.tsx` holds `activeSection` in `useState` and `switch`es over it in `renderContent()`. `mapRoleToSection(role, hrDepartment)` decides the landing section after login.
- Auth state = presence of `token` + `user` in `localStorage`. `App.tsx` reads them on mount; logout clears them.

### Per-module pattern (repeated 4–5×)
Each stage (Screening, Assessment, Selection, Engagement) follows the same shape:
1. `useXxxApplicants()` fetches `/api/applicants`, maps rows (snake_case → camelCase), filters by that module's allowed status set.
2. A table component renders the filtered list; a toolbar handles search/filter/add.
3. Row click opens `ApplicantSidebar` for detail/edit; status changes POST back and append to the module's `*_history` table.
4. Cross-module sync happens via `window` custom events (see [CONVENTIONS.md](CONVENTIONS.md) §Event bus).

## 6. Domain concepts

| Concept | Meaning |
|---------|---------|
| **Applicant** | A candidate in the recruitment pipeline (`recruitment_applicants`). Identified by free-text `applicant_no`. |
| **Principal** | The client company that hires deployed workers (formerly "Client"). 8 seeded: DATIAN, HOKEI, POBC, JINBOWAY, SURPRISE, THALESTE, AOLLY, ENJOY. An applicant ↔ principals is many-to-many via `applicant_principals`. |
| **Masterlist employee** | A deployed worker with full HR record (addresses, family, gov IDs, SBMA ID validity, transfer status). Lives in `masterlist_employees`, keyed by `fli_number`. |
| **Status** | The pipeline state machine. ~37 values; each module owns a subset (`ALLOWED_*_STATUSES` in each `utils`). |
| **History** | Append-only audit per stage: `screening_history`, `assessment_history`, `selection_history`, `engagement_history`. |

### Roles (`server/config/roles.js`)
`0 Admin · 1 Screening · 2 Assessment · 3 Selection · 4 Engagement · 5 Employee Relations`

> ⚠ Roles are enforced **client-side only** today, and the login response recomputes `role` from the department name rather than trusting the DB. See [IMPROVEMENTS.md](IMPROVEMENTS.md).

## 7. The Clients → Principals migration (important context)

The app was refactored from "Clients" to "Principals". Both names coexist as a compatibility layer:

| New (canonical) | Legacy alias (still wired) |
|-----------------|----------------------------|
| `principals` table | `clients` (SQL dump only, not in live DB) |
| `applicant_principals` | `applicant_clients` |
| `/api/principals`, `principalsController`, `principal.js` | `/api/clients`, `clientsController` (re-export), `client.js` (pool shim), `applicantClient.js` (re-export) |
| `src/api/principal.ts` | `src/api/client.ts` |

**When adding features, use the `principal` names.** Treat the `client` files/routes as deprecated shims to be removed once callers are migrated (tracked in [IMPROVEMENTS.md](IMPROVEMENTS.md)).

## 8. Data flow example — "advance an applicant's status"

1. User clicks a status control in a module table / `ApplicantSidebar`.
2. Hook optimistically updates local `users` state (and `selectedUser`).
3. Hook POSTs the full applicant payload (UPPER_CASE keys) to `/api/applicants`; controller upserts via `IFNULL` merge.
4. Hook POSTs a row to the module's `/api/applicants/<stage>-history`.
5. Hook dispatches `window` CustomEvent (`applicant-updated`); other mounted modules listen and reconcile their lists.
6. No rollback on failure today — the optimistic update stays even if the POST errors (see [IMPROVEMENTS.md](IMPROVEMENTS.md)).
