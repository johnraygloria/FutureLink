## `recruitment_applicants` (key columns)

Identity & personal: `id` (PK), `applicant_no` (free-text, indexed, **not unique**), `referred_by`, `last_name`, `first_name`, `ext`, `middle_name`, `gender`, `size`, `date_of_birth`, `date_applied`, `fb_name`, `age`, `location`, `contact_number`, `email`, `position_applied_for`, `experience`.

Legacy per-client columns (still present, superseded by `applicant_principals`): `datian`, `hokei`, `pobc`, `jinboway`, `surprise`, `thaleste`, `aolly`, `enjoy`.

Workflow: `status`, `requirements_status`, `final_interview_status`, `medical_status`, `status_remarks`, `applicant_remarks`.

Document flags (`tinyint(1)`, default 0): `recent_picture`, `psa_birth_certificate`, `school_credentials`, `nbi_clearance`, `police_clearance`, `barangay_clearance`, `sss`, `pagibig`, `cedula`, `vaccination_status`, `resume`, `coe`, `philhealth`, `tin_number`.

Timestamps: `created_at`, `updated_at`.

> Note: document fields are **booleans** here, but the frontend `ApplicantDocuments` type models several as strings. Mapping happens in the controllers/hooks (`toBit`, `toBitOrUndefined`).

## `principals` / `applicant_principals`

```
principals(id PK, name UNIQUE, created_at, updated_at)
applicant_principals(id PK, applicant_id, principal_id, created_at,
                     UNIQUE(applicant_id, principal_id))
```
- `applicant_id` references `recruitment_applicants.id`; `principal_id` references `principals.id`. **No FK constraints** — integrity is app-enforced.
- Seeded principals: DATIAN, HOKEI, POBC, JINBOWAY, SURPRISE, THALESTE, AOLLY, ENJOY (`initDefaultPrincipals`, `POST /api/principals/init-defaults`).

## `masterlist_employees` (deployed employees)

Wide HR table (~60 columns). Highlights:
- **Key:** `fli_number` (UNIQUE, generated by `getNextFliNumber`/`formatFliNumber`). Optional `applicant_no` links back to the recruit.
- Identity: `jbw_job_no`, `last_name`, `first_name`, `ext_name`, `middle_name`, `full_name` (derived), `principal`, `position`, `gender`, `date_hired`, `status` (default `ACTIVE`).
- Contact/links: `mobile_number`, `fb_link`, `email_address`, `sbma_id_validity`.
- Sizing: `shirt`, `shoes`; assessment: `level`, `level_remarks`.
- Gov IDs: `sss`, `pagibig`, `philhealth`, `tin`.
- Present address: `house_no`, `street`, `barangay`, `municipality`, `province`, `zip_code`, `complete_present`.
- Permanent address: `*_2` variants + `complete_address`.
- Family: `mothers_maiden_name`, `fathers_name`, `civil_status`, `spouses_name`, `num_children`, `children_ages`, `religion`.
- Emergency: `contact_person`, `contact_number`, `relation`.
- Lifecycle: `record_date`, `age`, `place`, `remarks`, `other_remarks`, `last_date_present`, `transfer_status`, `created_at`, `updated_at`.

**Self-migrating:** `masterlist.js` adds missing columns on boot (`migrateTable`, `columnExists`) and can `backfillPrincipalFromApplicants`. `syncDeployedApplicants()` (`POST /api/masterlist/sync`) imports deployed applicants into this table.

## History tables (all identical shape)

`screening_history`, `assessment_history`, `selection_history`, `engagement_history`:

```
id PK, applicant_no VARCHAR(50), action VARCHAR(100),
status VARCHAR(100), notes VARCHAR(255), created_at TIMESTAMP
```
- Joined to applicants on `TRIM(applicant_no) = TRIM(applicant_no)` (string join; no FK).
- Enriched read queries (`fetch*HistoryEnriched`) `LEFT JOIN recruitment_applicants` for name/position/date.
- Fallback: if `assessment_history` is empty, `getAssessmentHistory` derives rows from `screening_history` actions.

## `users`

```
id PK, email UNIQUE(nullable), hr_department UNIQUE, password_hash NOT NULL,
full_name, role TINYINT (0..5), created_at, updated_at
```
- One account per department; `password_hash` is bcrypt.
- Login **auto-creates** a missing department user, and (currently) has a `password === '1'` reset path — see the security section of [IMPROVEMENTS.md](IMPROVEMENTS.md).

## Data-integrity notes (for anyone extending the schema)

- `applicant_no` is the logical business key but is **not unique** and is generated app-side as `MAX(numeric applicant_no) + 1` — race-prone.
- There are **no foreign keys** anywhere; all relationships are app-maintained.
- Schema lives in **three** places that can disagree: the model `ensureTables()` DDL, `company_ftl.sql`, and the live DB. Prefer the live DB; converge the others.
